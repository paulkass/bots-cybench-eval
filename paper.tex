\begin{abstract}
Large language models are rapidly changing security work, but today's evaluations still lean toward offensive demonstrations: vulnerability scanning, exploit development, penetration testing, and CTFs.
Those tasks matter, yet best-case success rates at high test-time compute and in ideal harnesses miss the economic constraint shared by attackers and defenders: every reasoning step, tool call, and enrichment request spends budget.
We use Inspect to compare this cost-success tradeoff across offensive Cybench challenges and defensive Splunk BOTS v1 investigations.
The results show different scaling behavior for red- and blue-team tasks.
More compute can buy higher success on offensive CTFs, and scaled Chinese open-weight models can close much of the gap with frontier U.S. systems while remaining cost-competitive on red-team workloads.
SOC investigation, however, is less a pure reasoning race: effective defensive performance depends on disciplined tool use, telemetry navigation, and external-enrichment decisions.
We argue that security-agent evaluations should report not only success rate, but also cost, tool spend, and workflow fit.
Cost-aware, SOC-native benchmarks can better show where open models are already competitive and where defensive agents still need to improve.
\end{abstract}

\section{Introduction}

Security-agent evaluation is being pulled in two directions.
The public conversation is dominated by offensive demonstrations: models find bugs, write exploits, solve CTFs, and automate parts of penetration testing.
Those demonstrations are important because they measure real attacker capabilities.
They are incomplete because security operations centers (SOCs) ask a different question: can an agent navigate telemetry, preserve evidence, decide when enrichment is worth spending, and answer like an analyst under a budget?

This paper treats cost as part of the task rather than as a footnote.
Attackers and defenders both face a resource economy.
A red-team agent can sometimes spend more reasoning and sandbox time until a flag falls out.
A blue-team agent must spend searches, event drill-downs, and external enrichment calls only when they improve the investigation.
That distinction is the paper's organizing claim: offensive and defensive security tasks do not just have different labels; they reward different scaling behavior.

I compare two public benchmark families through an Inspect \citep{inspect_ai_2024} harness.
Cybench \citep{zhang2024cybench,inspect_evals_cybench} supplies offensive CTF-style tasks with shell and Python access inside challenge sandboxes.
Splunk Boss of the SOC (BOTS) v1 \citep{splunk_botsv1} supplies defensive incident-investigation questions over Splunk telemetry, with official competition points and hint penalties.
The goal is not a procurement leaderboard.
The source runs differ in cost limits, provider routing, concurrency, sandbox backend, and cost-accounting details.
The goal is narrower and more useful: report cost-success operating points, show where extra spend changes outcomes, and identify what a SOC-native evaluation must measure.

The main findings are:
\begin{itemize}
  \item On Cybench, additional per-sample budget can buy offensive progress: DeepSeek v4 Flash rises from 64.1\% pass@1 at a \$0.80 cap to 86.4\% at a \$2.10 cap.
  \item The strongest open-weight red-team operating point in these artifacts, DeepSeek v4 Flash at the higher cap, closes much of the gap to GPT-5.5 while remaining near it on cost per solve.
  \item On BOTSv1, the best defensive row is not the one that spends the most or calls the most tools: Claude Opus 4.8 reaches 93.9\% of BOTS points with 603 tool calls and \$13.24 in model-plus-priced-tool spend.
  \item Public SOC benchmarks need controls: no-tools BOTSv1 probes still score 50--75\% of points, so absolute scores are contamination-sensitive.
\end{itemize}

\section{Related Work}

Recent cyber evaluations test broad model risk, interactive execution, CTF-style problem solving, and exploit-oriented web-application tasks \citep{bhatt2024cyberseceval2,yang2023intercode,yang2023languagehackers,zhu2025cvebench}.
Cybench extends that line with professional CTF tasks in sandboxed environments \citep{zhang2024cybench}.
This paper uses Cybench as the offensive comparison point rather than as a proxy for all security work.

SOC work draws on a different body of practice.
Incident-response guidance and cybersecurity workforce taxonomies emphasize detection, analysis, evidence handling, response coordination, and operational judgment \citep{nist_sp800_61r3,nist_nice_800_181r1}.
A SOC analyst often needs to discover log schemas, write and revise SPL, inspect individual events, correlate artifacts, and decide whether an external provider call is justified.
Those behaviors are not captured by a flag-only CTF score.

\section{Benchmarks and Methodology}

All quantitative claims below were checked against the source evaluation repository at \path{~/projects/inspect-cyber-eval/}.
The source-of-truth artifacts are the generated chart tables and reports under \path{charts/}, the methodology notes in \path{charts/README.md}, the BOTS harness documentation in \path{bots/README.md}, and the raw Inspect logs listed by those artifacts.
No benchmark, Docker, Kubernetes, or Inspect workload was rerun for this rewrite.
The local deck \path{source-eval-repo/docs/actual-deck.odp} was used only for framing: application-security excitement, SOC aptitude as a distinct target, and the offense/defense scaling contrast.

\subsection{Cybench offensive runs}

The charted Cybench rows use the hard variant of \texttt{inspect\_evals/cybench}.
They run the source repository's autocompacting ReAct solver, expose only bash and Python tools with 180-second timeouts, use three epochs with a mean reducer, and score with the upstream \texttt{includes} accuracy metric.
Most rows use Docker sandboxes; the Claude Opus 4.8 row uses a Kubernetes sandbox to avoid local Docker subnet exhaustion.
The reported ``solved'' value is fractional pass@1: mean correctness over three epochs multiplied by the 39-challenge denominator.

\subsection{BOTS v1 defensive runs}

BOTS v1 is a public Splunk incident-investigation benchmark.
The source harness excludes the official warm-up question, leaving 31 scored questions worth 10,300 competition points.
Agents receive Splunk discovery/search/event tools plus limited VirusTotal, WHOIS, DNS, Brave Search, bash, and Python tools.
The prompt makes Splunk telemetry the primary evidence source and asks agents to use external enrichment selectively.
The primary metric is \texttt{bots\_points}: a correct answer earns official points minus hint penalties; an incorrect answer earns zero.
Binary \texttt{includes} accuracy is secondary.

The BOTS package also supports v2 and v3, but comparable paper-ready multi-model charts currently exist only for v1.
The v2/v3 artifacts are therefore harness coverage, not results used for model comparison.

\subsection{Cost accounting and interpretation}

Cybench costs are model-token costs from Inspect's usage ledger or reconstructed from token counts and registered rates where necessary.
BOTS costs start with Inspect model cost and add priced external enrichment calls visible in transcript tool events: Brave Search at \$0.005/request and non-cache WhoisXMLAPI WHOIS History credits converted at \$129 per 5,000 credits.
VirusTotal public API, DNS, and live WHOIS/RDAP are treated as zero marginal cost.
Infrastructure costs for Kubernetes, Splunk, storage, and analyst review time are excluded.

The analysis is descriptive.
These runs were not a randomized, controlled model-only experiment.
Cybench rows differ in cost caps, OpenRouter routing, concurrency, sandbox backend, retry-chain cost ledgers, and one challenge-set denominator.
BOTSv1 rows are more uniform at the task level, but provider defaults and reasoning settings still differ.
I therefore describe operating points and paired within-model changes, not universal model rankings.

\section{Results}

\subsection{Offensive CTFs: more budget can buy progress}

Table~\ref{tab:cybench} shows the current Cybench cost-success table.
GPT-5.5 is the top observed row at 94.1\% pass@1 and \$1.16 per solved-equivalent challenge.
The cleanest scaling signal is within one model family: DeepSeek v4 Flash improves from 64.1\% pass@1 at a \$0.80 cap to 86.4\% at a \$2.10 cap.
That higher-budget run solves 33.7 of 39 challenge-equivalents at \$1.45 per solve, close to GPT-5.5's 36.7 solved at \$1.16 per solve and above the Claude Opus rows on both success and cost per solve.

This is the quantitative basis for the open-weight red-team claim in the abstract.
The source repository records model slugs and providers rather than model-provenance metadata, so the benchmark evidence should be phrased as a cost-success result: the DeepSeek v4 Flash row closes much of the Cybench gap to GPT-5.5 under the tested harness.

\subsection{SOC investigations: tool volume does not explain success}

BOTS v1 changes the optimization problem.
The agent must navigate Splunk telemetry, use prerequisite case context, decide when to enrich externally, and submit precise answers under a point-and-hint scheme.
Table~\ref{tab:bots} shows the current full-agent BOTSv1 results.
Claude Opus 4.8 leads with 9,666.7 of 10,300 points (93.9\%) and 96.8\% binary accuracy.
It also has the best cost-success profile: \$13.24 in model-plus-priced-tool spend and 603 total tool calls across the 31 questions and three epochs.

GPT-5.5 reaches about 81\% of points, but spends \$47--49 and uses roughly 1,500 calls.
DeepSeek v4 Flash illustrates the difference from Cybench: raising its cap from \$2.10 to \$4.20 changes BOTSv1 points from 73.0\% to 73.9\%, a small and statistically inconclusive gain, while tool calls rise from 4,450 to 4,938.
In this defensive workload, more budget and more tool volume do not substitute for disciplined evidence gathering.
Because the no-tools controls below show substantial contamination, this is an observational tool-efficiency pattern rather than an isolated causal tool-use ablation.

\subsection{Public SOC benchmarks need decontamination controls}

BOTSv1 is useful because it provides real Splunk data and official scoring, but it is also public and old.
The source repository therefore includes a no-tools contamination probe.
The probe solver registers no Splunk, web, bash, Python, or other tools; it asks the model to answer from the question text, optional prerequisite context, and prior knowledge only.

The results are too high to ignore.
With prerequisite context enabled, Claude Opus 4.8 scores 7,700/10,300 points (74.8\%) and GPT-5.5 scores 6,400/10,300 (62.1\%) with zero tool events.
With prerequisite context disabled, Claude still scores 5,150 points (50.0\%) and GPT-5.5 scores 5,650 points (54.9\%).
These controls do not invalidate the harness, but they do invalidate naive claims that absolute BOTSv1 scores are pure evidence of live SOC investigation skill.

\section{Discussion}

\paragraph{A single security score hides the useful signal.}
The same model can occupy different parts of the cost-success frontier depending on whether the task is an offensive CTF or a SOC investigation.
Cybench rewards sustained sandbox reasoning and, for DeepSeek v4 Flash, additional per-sample budget.
BOTSv1 full-agent outcomes are most consistent with evidence navigation, answer precision, and restraint in tool use, though public-answer leakage prevents a clean causal attribution.
A model-selection process that reports only ``cyber score'' will miss this split.

\paragraph{Cost should be a first-class benchmark output.}
For CTFs, token spend and per-sample budget explain much of the practical tradeoff.
For SOC workflows, priced enrichment calls become part of the marginal cost of analysis.
The BOTS reports therefore add external-tool spend to model-token spend and report \$/1k BOTS points.
Those numbers still omit infrastructure and human review, but they are closer to the economic choice a security team faces than success rate alone.

\paragraph{Tool-call count is not tool competence.}
The best BOTSv1 row uses far fewer calls than the lower-scoring DeepSeek rows.
That does not mean fewer calls are always better; it means tool calls should be evaluated in context.
A SOC-native benchmark should ask whether the agent discovers the right fields, writes complete SPL, drills into representative events, validates an answer, and spends paid enrichment only when it changes the case.

\paragraph{Public benchmarks require controls.}
The no-tools BOTSv1 results show substantial memorization or public-answer leakage.
Future SOC evaluations should include no-tools probes, perturbed questions, answer-before-query checks, private holdouts, or fresh incident data before using absolute scores as capability evidence.
Public BOTS remains valuable as a reproducible harness and cost-accounting testbed, but not as a clean leaderboard by itself.

\section{Evaluation Recommendations}

A security-agent report should include at least five items:
\begin{enumerate}
  \item \textbf{Task-family labels.} Separate offensive exploitation, CTFs, alert triage, incident investigation, malware analysis, and enrichment-heavy workflows.
  \item \textbf{Workflow-native scoring.} Use pass@1 for CTF flags, competition points or analyst-grade rubrics for SOC questions, and report binary accuracy only as a secondary view when it drops information.
  \item \textbf{Cost accounting.} Report model-token spend, priced external-tool spend, cost per solve or per 1,000 points, and the pricing assumptions.
  \item \textbf{Tool-use evidence.} Report tool-call totals, cap curves, and representative transcripts or redacted traces that show whether calls advanced the investigation.
  \item \textbf{Decontamination controls.} For public benchmarks, include no-tools, no-context, perturbed-question, or private-holdout checks.
\end{enumerate}

\section{Limitations}

The experiments are observational.
They reuse existing Inspect logs rather than a newly randomized run matrix.
Cybench rows differ in cost limits, provider routing, concurrency, sandbox backend, and cost-accounting mechanics.
The DeepSeek v4 Pro Cybench row uses 38 challenges, not the usual 39.
BOTSv1 rows are more uniform, but provider defaults and reasoning settings still differ.
Bootstrap intervals in the appendix are descriptive robustness checks, not formal population-level claims.

The defensive analysis is BOTSv1-only.
The source repository packages BOTS v2 and v3, but comparable multi-model reports are not yet available.
BOTSv1 is public, answerable above chance without tools, and dependent on prerequisite context for many questions.
External-enrichment prices also drift across providers and enterprise contracts.
The paper's SOC conclusion should therefore be read as an evaluation-design finding, not as a final claim about production SOC readiness.

\section{Ethics, Artifacts, and Conclusion}

The experiments use public cybersecurity benchmarks and sandboxed environments.
This paper reports aggregate metrics and methodology only.
It does not publish raw private transcripts, secrets, API keys, or operational exploit instructions.
The appendix records provenance and robustness checks without copying raw \texttt{.eval} logs into this paper repository.

The central lesson is simple: security-agent evaluation should be cost-aware and workflow-specific.
Offensive CTFs and defensive SOC investigations stress different behaviors and show different scaling patterns.
More compute can improve CTF success, and an open-weight DeepSeek row demonstrates a credible red-team cost-success operating point.
SOC work, however, depends on disciplined tool use, telemetry navigation, and enrichment decisions, and public SOC benchmarks must be decontaminated before their absolute scores are trusted.

\section{Uncertainty Method}

The paper reports descriptive robustness intervals.
For Cybench, each bootstrap replicate resamples challenge IDs with replacement and retains the three epochs for each selected challenge.
For BOTSv1, each replicate resamples question IDs with replacement and retains the three epochs and official point weights for each selected question.
Paired contrasts resample only task IDs common to both compared logs.

\section{BOTSv1 Sequential-Context Check}

The BOTSv1 task excludes the official warm-up question and evaluates 31 scored questions.
Twenty-three of those questions have explicit dependencies on earlier case facts and receive prerequisite context in the prompt.
This matches the benchmark's sequential investigation structure, but it should be interpreted as follow-up investigation with known case state rather than cold-start reconstruction.

For Claude Opus 4.8, independent questions score 1,445/1,500 points (96.3\%) and dependent questions score 8,221.7/8,800 points (93.4\%).
The standard GPT-5.5 row scores 1,345/1,500 points (89.7\%) on independent questions and 7,000/8,800 points (79.5\%) on dependent questions.
Claude's mean-reduced point loss decomposes into 133.3 points from hint penalties and 500 points from misses; GPT-5.5 loses 205 points to hint penalties and 1,750 points to misses.
