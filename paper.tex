\begin{abstract}
Large language models are rapidly changing security work, but today's evaluations still lean toward offensive demonstrations: vulnerability scanning, exploit development, penetration testing, and CTFs.
Those tasks matter, yet best-case success rates at high test-time compute and in ideal harnesses miss the economic constraint shared by attackers and defenders: every reasoning step, tool call, and enrichment request spends budget.
We use Inspect to compare this cost-success tradeoff across offensive Cybench challenges and defensive Splunk BOTS v1 investigations.
The results show different scaling behavior for red- and blue-team tasks.
More compute can buy higher success on offensive CTFs, and scaled Chinese open-weight models can close much of the gap with frontier U.S. systems while remaining cost-competitive on red-team workloads.
SOC investigation, however, is less a pure reasoning race: effective defensive performance depends on disciplined tool use, telemetry navigation, and external-enrichment decisions.
We argue that security-agent evaluations should report not only success rate, but also cost, tool spend, and workflow fit.
Cost-aware, SOC-native benchmarks can better show where open models are already competitive and where defensive agents still need to improve.
\end{abstract}

cybersecurity benchmarks, AI agents, security operations centers, Cybench, BOTS, Inspect, cost accounting

\section{Introduction}
\label{sec:introduction}

Security-agent evaluations increasingly report impressive offensive demonstrations: agents solve capture-the-flag (CTF) tasks, find vulnerabilities, write exploits, and automate pieces of penetration testing.
These tasks matter because they measure capabilities adversaries can also use.
They are not enough for applied security operations.
A security operations center (SOC) asks whether an agent can navigate telemetry, preserve evidentiary context, decide when enrichment is worth buying, and return an analyst-grade answer under a budget.

This paper treats cost as part of the task rather than an afterthought.
Both attackers and defenders operate under a resource economy: each reasoning step, shell command, Splunk search, and paid enrichment call consumes time or money.
The central claim is that offensive and defensive security tasks reward different scaling behavior.
In CTF-style tasks, extra per-sample budget can buy more exploration and higher success.
In SOC investigation, tool volume alone is a poor proxy for competence; success depends on using the right telemetry and enrichment at the right time.

We compare offensive Cybench challenges \citep{zhang2024cybench,inspect_evals_cybench} and defensive Splunk Boss of the SOC (BOTS) v1 investigations \citep{splunk_botsv1} using Inspect \citep{inspect_ai_2024}.
Cybench provides sandboxed CTF-style tasks with shell and Python access.
BOTS v1 provides public Splunk telemetry and official competition questions with point values and hint penalties.
The available runs differ in cost limits, provider routing, concurrency, sandbox backend, and cost-accounting details.
We characterize cost-success operating points, identify which comparisons are supported by paired evidence, and derive requirements for SOC-native security-agent benchmarks.

Our contributions are as follows:
\begin{itemize}
  \item A cost-aware comparison of offensive Cybench and defensive BOTSv1 operating points using the same evaluation framework;
  \item Evidence that additional per-sample budget can materially improve an offensive Cybench row, while a comparable BOTSv1 budget increase produces little defensive gain;
  \item Model-token plus priced-tool cost accounting for SOC investigation, including cost per 1,000 BOTS points;
  \item No-tools BOTSv1 controls showing that public SOC benchmarks require decontamination checks before absolute scores are interpreted as live investigation skill.
\end{itemize}

\section{Related Work}
\label{sec:related-work}

Cybersecurity model evaluations often emphasize offensive or execution-heavy tasks.
CyberSecEval 2 measures a broad set of cyber risks and capabilities \citep{bhatt2024cyberseceval2}; InterCode studies interactive code execution \citep{yang2023intercode}; Language Agents as Hackers and CVE-Bench evaluate CTF and exploitation-oriented behavior \citep{yang2023languagehackers,zhu2025cvebench}.
Cybench extends this line with professional CTF tasks in sandboxed environments \citep{zhang2024cybench}.
These evaluations are valuable, but they do not directly measure the operational economics of alert triage, incident investigation, or enrichment-heavy SOC work.

SOC analysis draws on a different body of practice.
Incident-response guidance and cybersecurity workforce taxonomies emphasize detection, analysis, evidence handling, response coordination, and operational judgment \citep{nist_sp800_61r3,nist_nice_800_181r1}.
An analyst must discover log schemas, write and revise SPL, inspect representative events, correlate artifacts, and decide whether external enrichment will change the case.
A flag-only score or a raw tool-call count cannot distinguish those behaviors.

This paper connects those two evaluation cultures.
It keeps offensive CTF success rates, but adds cost-per-solve and paired budget comparisons.
It keeps BOTS competition points, but adds priced-tool accounting, tool-call efficiency views, and no-tools contamination probes.

\section{Evaluation Design}
\label{sec:evaluation-design}

All reported numbers come from pre-existing Inspect logs and generated chart artifacts in the source evaluation repository.
The paper uses logical artifact paths such as \path{source-eval-repo/charts/cybench_cost_table.csv} to preserve provenance without exposing local paths or copying raw logs into the manuscript repository.

\subsection{Offensive benchmark: Cybench}
\label{sec:cybench-method}

The Cybench runs use the hard variant of \texttt{inspect\_evals/cybench}.
The charted rows use the source repository's autocompacting ReAct solver \citep{yao2023react}; expose only \texttt{bash(timeout=180)} and \texttt{python(timeout=180)}; run three epochs with a mean reducer; and score with the upstream \texttt{includes} accuracy metric.
Most rows use Docker sandboxes.
The Claude Opus 4.8 row uses a Kubernetes sandbox to avoid local Docker subnet exhaustion.
The reported solved count is fractional pass@1: mean correctness over three epochs multiplied by the 39-challenge denominator.

The Cybench rows are not a fully controlled model-only experiment.
Per-sample cost limits, OpenRouter provider routing, concurrency, sandbox backend, and some retry-cost ledgers differ across rows.
DeepSeek v4 Pro uses 38 challenges because one challenge was excluded after a deterministic sandbox-initialization failure; the generated chart normalizes its solved-equivalent value to the 39-challenge denominator.

\subsection{Defensive benchmark: BOTS v1}
\label{sec:bots-method}

BOTS v1 is a public Splunk incident-investigation benchmark.
The source harness excludes the official warm-up question, leaving 31 scored questions worth 10,300 official competition points.
Agents receive Splunk discovery, search, and event-inspection tools plus limited VirusTotal, WHOIS, DNS, Brave Search, bash, and Python tools.
The prompt makes Splunk telemetry the primary evidence source and asks agents to use external enrichment selectively.

The primary metric is \texttt{bots\_points}: a correct answer earns official points minus hint penalties; an incorrect answer earns zero.
Binary \texttt{includes} accuracy is reported as a secondary view.
Rows use three epochs and mean reduction, so point totals are reduced back to the single-epoch 10,300-point scale.
Twenty-three of the 31 scored questions receive prerequisite case context from earlier official questions, matching the sequential investigation structure but making the task closer to follow-up analysis than cold-start incident reconstruction.

The source package also supports BOTS v2 and v3.
Comparable multi-model reports currently exist only for v1, so v2/v3 are treated as harness coverage rather than evidence for model comparison.

\subsection{Cost accounting}
\label{sec:cost-accounting}

Cybench costs are model-token costs from Inspect's usage ledger or reconstructed from token counts and registered rates where needed.
BOTS costs start with Inspect model cost and add priced external enrichment calls visible in transcript tool events.
The source reports charge Brave Search at \$0.005 per request and non-cache WhoisXMLAPI WHOIS History credits at a user-provided paid-plan conversion of \$129 per 5,000 credits.
VirusTotal public API, DNS, and live WHOIS/RDAP are treated as zero marginal cost.
Infrastructure costs for Kubernetes, Splunk, storage, and analyst review time are excluded.

Because these are descriptive artifacts, the analysis avoids universal model rankings.
We emphasize operating points and paired within-model comparisons where the evidence supports them.

\subsection{Public SOC benchmarks need contamination controls}
\label{sec:contamination}

BOTSv1 is valuable because it supplies real Splunk data and official scoring, but it is also public and old.
The source repository therefore includes a no-tools probe.
The probe solver registers no Splunk, web, bash, Python, or other tools; it asks the model to answer from the question text, optional prerequisite context, and prior knowledge only.

The results are too high to treat absolute BOTSv1 scores as clean evidence of live investigation skill.
With prerequisite context enabled, Claude Opus 4.8 scores 7,700/10,300 points (74.8\%) and GPT-5.5 scores 6,400/10,300 (62.1\%) with zero tool events.
With prerequisite context disabled, Claude still scores 5,150 points (50.0\%) and GPT-5.5 scores 5,650 points (54.9\%).
These controls do not make BOTS useless; they make decontamination controls mandatory.

\section{Discussion: What a SOC-Native Evaluation Should Measure}
\label{sec:discussion}

A single ``cyber score'' hides the useful signal.
The same model family can occupy different parts of the cost-success frontier depending on whether the task is an offensive CTF or a SOC investigation.
Cybench rewards sustained sandbox reasoning and, for DeepSeek v4 Flash, additional per-sample budget.
BOTSv1 rewards a different workflow: finding the relevant telemetry, validating answers, and avoiding unproductive tool churn.
Because public-answer leakage is substantial, the defensive interpretation should be framed as an evaluation-design finding rather than a final production-readiness ranking.

Cost should be a first-class benchmark output.
For CTFs, token spend and per-sample budget explain much of the practical tradeoff.
For SOC workflows, paid enrichment calls become part of the marginal cost of analysis.
The BOTS reports therefore add external-tool spend to model-token spend and report dollars per 1,000 BOTS points.
Those numbers still omit infrastructure and human review, but they are closer to the economic choice a security team faces than success rate alone.

Tool-call count is not tool competence.
The best BOTSv1 row uses far fewer calls than lower-scoring rows, but fewer calls are not inherently better.
A SOC-native benchmark should ask whether the agent discovers the right fields, writes complete SPL, drills into representative events, validates the answer, and spends paid enrichment only when it changes the case.

Public benchmarks require controls.
No-tools, no-context, perturbed-question, answer-before-query, private-holdout, or fresh-incident checks should accompany any public SOC benchmark before absolute scores are interpreted as capability evidence.
BOTS remains useful as a reproducible harness and cost-accounting testbed; it should not be used as a clean leaderboard by itself.

\section{Reporting Recommendations}
\label{sec:recommendations}

Security-agent reports should include at least five items:
\begin{enumerate}
  \item \textbf{Task-family labels.} Separate offensive exploitation, CTFs, alert triage, incident investigation, malware analysis, and enrichment-heavy workflows.
  \item \textbf{Workflow-native scoring.} Use pass@1 for CTF flags, competition points or analyst-grade rubrics for SOC questions, and report binary accuracy only when it does not hide point weight or hint penalties.
  \item \textbf{Cost accounting.} Report model-token spend, priced external-tool spend, cost per solve or per 1,000 points, and pricing assumptions.
  \item \textbf{Tool-use evidence.} Report tool-call totals, cap curves, and representative redacted traces that show whether calls advanced the investigation.
  \item \textbf{Decontamination controls.} For public benchmarks, include no-tools, no-context, perturbed-question, answer-before-query, or private-holdout checks.
\end{enumerate}

\section{Limitations}
\label{sec:limitations}

The experiments are observational.
They reuse existing Inspect logs rather than a newly randomized run matrix.
Cybench rows differ in cost limits, provider routing, concurrency, sandbox backend, and cost-accounting mechanics.
The DeepSeek v4 Pro Cybench row uses 38 challenges, not the usual 39.
BOTSv1 rows are more uniform at the task level, but provider defaults and reasoning settings still differ.
Bootstrap intervals in Appendix~\ref{app:uncertainty} are descriptive robustness checks, not formal population-level claims.

The defensive analysis is BOTSv1-only.
The source repository packages BOTS v2 and v3, but comparable multi-model reports are not yet available.
BOTSv1 is public, answerable above chance without tools, and dependent on prerequisite context for many questions.
External-enrichment prices also drift across providers and enterprise contracts.
The SOC conclusion should therefore be read as an evaluation-design result, not as a final claim about production SOC readiness.

\section{Artifact and Ethical Considerations}
\label{sec:ethics}

The experiments use public cybersecurity benchmarks and sandboxed environments.
This paper reports aggregate metrics and methodology only.
It does not publish raw private transcripts, secrets, API keys, or operational exploit instructions.
The manuscript repository keeps only figures, tables, and provenance notes needed to audit aggregate claims; raw \texttt{.eval} logs remain in the separate source evaluation repository and should undergo a sanitization pass before any public artifact release.

\section{Conclusion}
\label{sec:conclusion}

Security-agent evaluation should be cost-aware and workflow-specific.
Offensive CTFs and defensive SOC investigations stress different behaviors and show different scaling patterns.
More compute can improve CTF success, and the DeepSeek v4 Flash Cybench row demonstrates a competitive red-team cost-success operating point under the tested harness.
SOC work, however, depends on disciplined tool use, telemetry navigation, and enrichment decisions.
Public SOC benchmarks remain valuable, but no-tools BOTSv1 controls show that their absolute scores require decontamination checks before they are used as capability claims.

\section{Uncertainty Method}
\label{app:uncertainty}

The paper reports descriptive bootstrap intervals.
For Cybench, each replicate resamples challenge IDs with replacement and retains the three epochs for each selected challenge.
For BOTSv1, each replicate resamples question IDs with replacement and retains the three epochs and official point weights for each selected question.
Paired contrasts resample only task IDs common to both compared logs.

\section{BOTSv1 Sequential-Context Check}
\label{app:bots-context}

The BOTSv1 task excludes the official warm-up question and evaluates 31 scored questions.
Twenty-three of those questions have explicit dependencies on earlier case facts and receive prerequisite context in the prompt.
This matches the benchmark's sequential investigation structure, but it should be interpreted as follow-up investigation with known case state rather than cold-start reconstruction.

For Claude Opus 4.8, independent questions score 1,445/1,500 points (96.3\%) and dependent questions score 8,221.7/8,800 points (93.4\%).
The standard GPT-5.5 row scores 1,345/1,500 points (89.7\%) on independent questions and 7,000/8,800 points (79.5\%) on dependent questions.
Claude's mean-reduced point loss decomposes into 133.3 points from hint penalties and 500 points from misses; GPT-5.5 loses 205 points to hint penalties and 1,750 points to misses.
