\begin{abstract}
AI security evaluations often emphasize exploit-finding, application-security, or capture-the-flag tasks, yet security operations center (SOC) work asks a different question: can an agent navigate telemetry, pivot through evidence, use enrichment sparingly, and answer incident questions with analyst discipline?  I report experiments from an Inspect-based evaluation harness that runs contemporary models on two public benchmark families: Cybench hard-variant CTF challenges and Splunk Boss of the SOC (BOTS) v1 investigations.  The headline result is not a single universal model ranking.  On Cybench, GPT-5.5 reached 94.1\% pass@1 and DeepSeek v4 Flash improved from 64.1\% to 86.4\% when its per-sample budget was raised, suggesting that offensive CTF performance can benefit substantially from additional compute/tool budget.  On BOTSv1, however, Claude Opus 4.8 earned 9,666.7/10,300 competition points (93.9\%) with 603 tool calls and \$13.24 in model-plus-priced-tool spend, while GPT-5.5 earned roughly 81\% of points with more than twice the tool calls and over three times the cost.  These results support a practical evaluation message: offensive and defensive security tasks have different scaling behaviors, and SOC aptitude should be benchmarked with SOC-native tools, scoring, and cost accounting rather than inferred from CTF performance alone.
\end{abstract}

\section{Introduction}

The current discussion around AI for security is strongly shaped by application-security demonstrations: models find bugs, reason about code, and sometimes exploit vulnerable services.  Those capabilities matter, but they do not exhaust the work of security teams.  Incident-response guidance and cybersecurity workforce taxonomies emphasize detection, analysis, evidence handling, and response coordination as distinct work from vulnerability discovery \citep{nist_sp800_61r3,nist_nice_800_181r1}.  A SOC analyst spends much of a shift formulating searches over telemetry, validating weak signals, correlating host/network/application evidence, and deciding when an external enrichment call is worth its cost and latency.  A model that is strong at an offensive CTF may not be the model that best performs defensive incident investigation.

This paper describes a set of experiments in a local evaluation repository, \path{~/projects/inspect-cyber-eval/}, and the paper-writing artifacts in the present repository.  The experiments use Inspect \citep{inspect_ai_2024}, a task/solver/scorer framework with sandboxed tools, to run two public benchmark families.  Cybench \citep{zhang2024cybench,inspect_evals_cybench} represents offensive, CTF-style tasks: agents work in per-challenge sandboxes with shell/Python tools and submit flags.  Splunk Boss of the SOC (BOTS) \citep{splunk_botsv1,splunk_botsv2,splunk_botsv3} represents defensive, SOC-style questions over pre-indexed Splunk telemetry.  The BOTS harness packages the public datasets as Inspect tasks, gives agents Splunk and enrichment tools, and scores with the original competition point economy rather than only exact-match accuracy.

The experiments are best read as an engineering-and-evaluation study rather than as a perfectly controlled model bake-off.  The source repository's chart audit records several non-invariants: per-sample cost limits, OpenRouter provider routing, challenge-set size for one Cybench row, concurrency, and provider-specific cost accounting differed across runs.  I therefore avoid claims such as ``model A is intrinsically better than model B'' unless the comparison is especially direct.  Instead, I emphasize patterns that matter for security evaluation design:

\begin{itemize}
  \item Offensive CTF and defensive SOC tasks reward different behavior.  Cybench rewards sustained exploration of challenge sandboxes; BOTS rewards targeted evidence gathering and disciplined tool use.
  \item More compute/tool budget helped some offensive runs.  The clearest ablation is DeepSeek v4 Flash on Cybench, where raising the per-sample cap from \$0.80 to \$2.10 raised pass@1 from 64.1\% to 86.4\%.
  \item More tool use was not automatically better for SOC.  In BOTSv1, the best run used the fewest total tool calls among the compared full runs, suggesting that defensive benchmark performance depends heavily on tool quality and tool-selection policy.
  \item Cost should be reported in the units that matter for the task family.  For Cybench, I report model-token spend per solve.  For BOTS, I also include priced external enrichment calls because SOC workflows routinely pay for search, reputation, and WHOIS-style lookups.
\end{itemize}

\section{Benchmarks and Harness}

\subsection{Cybench as an offensive CTF benchmark}

Cybench was introduced as a framework for evaluating language-model agents on professional CTF cybersecurity tasks \citep{zhang2024cybench}.  The local experiments used the Inspect Evals implementation of \texttt{inspect\_evals/cybench}.  In the evaluated logs, each full hard-variant run contains 39 challenge instances and was run with three epochs and a mean reducer, so the reported pass@1 can be fractional.  The agent scaffold for the charted ``+ac'' rows was \path{agents/react_autocompact.py@react_autocompact}: an upstream Cybench ReAct prompt \citep{yao2023react}, bash and Python tools with 180-second timeouts, a three-attempt loop, Inspect automatic compaction, and a one-refusal abort rule.  The Cybench sandbox was Docker.

The headline metric is the upstream \texttt{includes} accuracy: a challenge epoch is correct if the submitted answer includes the target flag.  I report ``solved'' as the mean-correct equivalent out of 39 challenges.  For example, 36.7 solved means that, across 39 challenges and three epochs, the model's mean success rate is equivalent to 36.7 challenges solved at pass@1.

\subsection{BOTS as a defensive SOC benchmark}

BOTS is Splunk's Boss of the SOC competition data.  Version 1 centers on the Po1s0n1vy/APT web compromise and Cerber ransomware scenario against Wayne Enterprises.  The source repository packages BOTS v1, v2, and v3 as one parameterized Inspect task, but the results in this paper focus on BOTSv1 because the available charted comparison is a complete, audited multi-model set.

For full BOTSv1 runs, the task excludes the official warm-up question 1, leaving 31 scored benchmark questions.  Each question is an Inspect sample over a per-sample Splunk sandbox.  The agent receives Splunk discovery/search/event tools, limited external enrichment tools (VirusTotal, WhoisXMLAPI, Brave Search, DNS, WHOIS), and guarded bash/Python tools.  The BOTS runs reported here used the Kubernetes sandbox path recorded in the logs: the local Inspect driver launched per-sample pods in the approved non-prod context with the BOTS data mounted read-only in the sandbox.  No additional experiments were launched while drafting this paper.

BOTS is not naturally a binary benchmark.  The original competition assigns base points by question difficulty and reveals hints after incorrect submissions, with point penalties.  The harness therefore uses \texttt{bots\_points} as the primary scorer: a correct answer earns \(\max(0,\text{base points}-\text{hint penalty})\), and a wrong answer earns 0.  I also report the secondary \texttt{includes} binary accuracy because it is useful for comparison, but BOTS points are the headline SOC metric.

\section{Methodology}

All quantitative claims in this paper were checked against source artifacts in \path{~/projects/inspect-cyber-eval/}.  The generated tables and figures come from \path{charts/*.html}, \path{charts/*.csv}, and \path{charts/*.png}; the raw Inspect logs in \path{logs/*.eval} are treated as the highest-fidelity source for model configuration, samples, costs, and transcripts.  The most important source files are listed in Appendix~\ref{app:audit}.

\paragraph{Cost accounting.}  Cybench costs are model-token costs from Inspect's model-usage ledger when available, with provider-specific caveats recorded in \path{charts/README.md}.  OpenRouter and OpenAI rows use Inspect's ledger; Anthropic costs were computed from token counts and registered rates where the API did not populate \texttt{total\_cost}.  BOTSv1 costs start with Inspect model costs and add priced external enrichment tools when a charged provider call appears in the transcript.  The tool-cost report charges Brave Search at \$0.005/request, treats VirusTotal public API use, DNS, and live WHOIS as zero marginal cost, and converts WhoisXMLAPI DRS credits to USD at the user-provided paid-plan rate \$129/5,000 credits.

\paragraph{Resource curves.}  The source charts include cost-budget curves and tool-call cap curves.  For Cybench and BOTS, the tool-call cap curve asks: if each sample epoch were allowed at most \(X\) non-submit tool calls, what share of sample epochs would still be solved?  Sample epochs that originally required more than \(X\) calls are counted as unsolved.  For BOTS, the cost-budget chart has both model-token-only and model-plus-priced-tool curves.

\paragraph{Threats to direct comparison.}  The charted rows are not a fully controlled model-only experiment.  On Cybench, per-sample cost limits varied, OpenRouter provider routing differed for some rows, DeepSeek v4 Pro used 38 rather than 39 challenges because \texttt{chunky} had a deterministic sandbox initialization failure, and same-day failed retries are included in some cost totals.  On BOTSv1, the rows are more uniform at the task level (all are full v1, three epochs, mean reducer, message limit 250, Kubernetes sandbox), but model defaults such as reasoning effort still differ by provider and row.  These caveats are central to the interpretation: the results support benchmark-design and task-family claims more strongly than leaderboard claims.

\section{Results}

This is the cleanest budget-scaling signal in the Cybench data: holding the model and core solver fixed, raising the cap to \$2.10 moved the run from 25.0 to 33.7 solved-equivalent challenges.

The tool-call data complicates the naive ``more actions equals better'' story.  GPT-5.5 solved the most challenges with a mean of 20.6 non-submit tool events per sample epoch. The result is better summarized as: on offensive CTF tasks, additional budget can enable longer exploration and recovery from dead ends, but model competence and refusal behavior still dominate.

The BOTS result is the strongest evidence for the paper's defensive-evaluation message.  In a SOC setting, the agent's job is not to exhaustively enumerate every possible SPL query.  It must choose discriminating searches, inspect the returned fields, pivot through the incident chain, and reserve external enrichment calls for high-value indicators.  The best BOTSv1 run appears to do more with fewer actions.  That is exactly the failure mode hidden by an offense-only benchmark: an agent can look capable when the objective is to keep trying until a flag falls out, yet be inefficient or brittle when the task is evidence triage under scoring and tool constraints.

\section{Discussion}

\paragraph{Do not extrapolate SOC readiness from CTF results.}  Cybench is valuable because it measures autonomous challenge solving in realistic sandboxes.  But its task shape is closer to offensive exploitation and reverse engineering than to SOC work.  BOTS asks for a different bundle of skills: familiarity with log schemas, SPL query formulation, hypothesis pruning, external enrichment, and answer formatting under a point-and-hint economy.  The model ordering changes across these task families, which means an organization selecting a model for security operations should evaluate on SOC-native workloads.

\paragraph{Compute scaling and tool scaling differ by task family.}  The DeepSeek v4 Flash Cybench ablation is a concrete example where additional per-sample budget unlocked more solves.  By contrast, in BOTSv1, raising DeepSeek v4 Flash's cap from \$2.10 to \$4.20 moved points only from 7,518.3 to 7,610.0 while increasing total tool calls from 4,450 to 4,938.  In this dataset, defensive performance was not primarily a matter of allowing more calls.  It depended on whether the model could spend calls well.

\paragraph{Tool-cost accounting changes the evaluation question.}  In offensive CTFs, model token spend is often the dominant marginal cost.  In SOC workflows, external data providers can be part of the actual marginal cost of analysis.  The BOTS report therefore separates model cost, tool cost, and combined cost.  This distinction matters: some runs look inexpensive on model tokens but rely on paid WHOIS history or search calls; others are more selective.  A production SOC evaluation should make that accounting explicit before optimizing prompts or selecting a model.

\paragraph{The harness is itself a research contribution.}  The BOTS implementation is not just a chart.  It packages three public BOTS datasets as a single Inspect task, uses one lean data-free Splunk image, supports Docker and Kubernetes sandboxes, excludes the warm-up from benchmark scoring, restores prerequisite context for questions that refer to previous answers, and preserves competition-style scoring.  These engineering choices make the benchmark more faithful to the original SOC game while keeping model transcripts and costs auditable in Inspect logs.

\section{Limitations}

First, the experiments were not a randomized, fully controlled model-only benchmark.  Cybench rows differ in cost limits, provider routing, concurrency, and in one case challenge count.  Same-day failed retries are included in selected cost rows.  The paper therefore treats the results as evidence about task-family behavior and evaluation design, not as final public rankings of proprietary models.

Second, the model names and provider routes reflect the local experimental environment.  Some rows use future or provider-specific model identifiers, and OpenRouter routes may aggregate multiple providers.  Readers should reproduce the harness with their own model endpoints before making procurement decisions.

Third, the BOTSv1 analysis covers only version 1.  The source harness supports BOTS v2 and v3, and the repository contains additional logs, but the complete multi-model report used here is BOTSv1.  The v2 and v3 datasets differ in scenario, size, and question dependencies; they should be analyzed before claiming full SOC generality.

Fourth, BOTS includes live or time-sensitive enrichment tasks.  Whois and malware-reputation data can drift relative to the original competition era.  The harness mitigates some dependency issues by appending prerequisite context, but external data drift remains a threat to validity and should be tracked question-by-question.

Finally, cost numbers are useful but imperfect.  Cybench costs are model-token costs with provider-specific ledger caveats.  BOTS adds priced tool calls only where the transcript shows a charged provider API call and where a pricing model was available.  These numbers are good enough to compare orders of magnitude and to motivate cost-aware evaluation, but they should not be read as universal production costs.

\section{Ethics and Safety}

The experiments use public cybersecurity benchmarks and sandboxed environments.  Cybench challenges run in per-challenge containers; BOTS runs against local or Kubernetes Splunk sandboxes with pre-indexed public datasets.  The paper does not publish raw transcripts, secrets, API keys, or exploit instructions beyond high-level benchmark descriptions.  No additional benchmark runs, Docker workloads, or Kubernetes experiments were launched for this draft.  If future experiments are added, they should follow the source repository's operational constraints: run from a checked-in repository state, use the approved research Kubernetes namespace for new k8s work, cap spend, and preserve Inspect logs for audit.

\section{Conclusion}

The central finding is simple: not all security tasks are the same.  Offensive CTF benchmarks and defensive SOC investigations stress different agent skills and show different scaling behavior.  In the experiments reported here, additional compute helped an offensive Cybench run substantially, while the strongest BOTSv1 SOC run won by using tools more selectively and cheaply.  Evaluation programs should therefore stop treating cybersecurity as one scalar capability.  They should report task-family-specific results, use scoring that matches the real workflow, and account for the full cost of model tokens and external tools.  For SOC adoption in particular, the right benchmark is not ``can the model solve CTFs?'' but ``can the model investigate incidents like an analyst?''
